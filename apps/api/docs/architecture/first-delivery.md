# First Delivery Overview

This document is the first standalone delivery note for the `apps/api` PoC. It summarizes the current scope, runtime design, exposed behavior, operational setup, and implementation trade-offs without changing the execution-oriented README.

## Purpose and Scope

- The PoC exposes a product-grounded chat API for a catalog dataset.
- The current deployable unit is `apps/api`.
- The implemented scope is intentionally narrow: a dataset-backed `POST /chat` flow plus basic service and health routes.
- The goal of this delivery is to prove grounded retrieval and response shaping against the provided product catalog, not to provide a general-purpose conversational assistant.

### Known assignment deviation

- The original assignment asks for a RAG pipeline that uses an LLM to synthesize the final answer from retrieved product evidence.
- The current PoC does not implement LLM-based synthesis.
- Instead, it retrieves dataset evidence and produces a deterministic/template-based answer from that evidence.

## High-Level Design

### Runtime shape

`HTTP request -> FastAPI route -> application use case -> dataset-backed retrieval -> deterministic answer shaping -> JSON response`

### Main building blocks

| Area | Current role |
| --- | --- |
| `main.py` | FastAPI bootstrap and application wiring |
| `src/domain` | Core request/result concepts and shared text normalization |
| `src/application` | Chat use-case orchestration |
| `src/infrastructure/input/http` | HTTP transport adapter for route handling |
| `src/infrastructure/output` | Dataset-backed retrieval and row mapping |

### Retrieval and response strategy

- The API reads product data from `data/product_catalog_dataset.json` unless `CATALOG_DATASET_PATH` overrides it.
- Retrieval is scoped by `site_id` before candidate products are returned.
- Matching is lexical and uses normalized catalog text fields.
- The response generator is deterministic/template-based and only uses retrieved catalog context.
- No embeddings, vector database, reranker, or LLM inference are part of the current implementation.

## API Behavior Summary

| Route | Behavior |
| --- | --- |
| `GET /` | Returns service status metadata |
| `GET /health` | Returns process/liveness status |
| `POST /chat` | Accepts `site_id` and `query`, retrieves site-scoped catalog evidence, and returns a grounded answer |

### `POST /chat` response contract

- Success responses return:
  - `answer`
  - `retrieved_products`
- Each `retrieved_products` item includes:
  - `article_id`
  - `product_id`
  - `variant_id`
  - `title`
  - `summary`
  - `category`
  - `site_id`
  - `score`

### Current runtime guarantees

- Dataset failures are surfaced as sanitized `503` readiness errors instead of opaque `500` responses.
- Summary content is normalized before returning it to clients.
- Off-topic or no-result queries do not invent products.

## Decisions and Trade-Offs

| Decision | Why it was chosen | Trade-off |
| --- | --- | --- |
| Lexical retrieval over embeddings | Small PoC scope, simple runtime, no external search infrastructure | Lower semantic recall for paraphrases and fuzzy intent |
| Deterministic/template answer generation | Keeps responses explainable and grounded in retrieved catalog evidence | Less natural phrasing and no open-ended reasoning |
| Dataset file as the only source of truth | Fast to inspect, easy to run locally, no database dependency | Limited scalability and weaker operational flexibility |
| `site_id` scoping before retrieval output | Enforces tenant/site separation in the current dataset model | Cross-site discovery is intentionally unsupported |
| Sanitized `503` readiness failures | Clearer operational behavior when dataset loading is broken | Failure detail is intentionally limited to protect internals |

## Known Limitations and Deferred Work

- The implementation does not satisfy the original assignment's LLM-based answer synthesis requirement; answers are deterministic/template-based rather than generated by an LLM over retrieved evidence.
- Retrieval is lexical only, so semantic paraphrases and fuzzier intent can be missed.
- Ranking remains simple and dataset-bound; there is no embeddings/vector search layer, reranker, or retrieval quality evaluation pipeline.
- The dataset file is the only runtime data source; there is no more operationally robust backing store in the current PoC.
